# 六部执行架构设计概要

> 三省六部制的执行层设计，作为现有三省 SKILL 的扩展提案。
> 当前状态：设计阶段，待确认后实现。

---

## 一、六部在三省流程中的位置

```
用户请求 → 尚书省（S0/S1）
         → 中书省（规划）→ 门下省（审核）→ 尚书省（调度）
                                              │
                           ┌──────────────────┤
                           │ 有 department     │ 无 department
                           ▼                  ▼
                       六部 subagent      尚书省自行执行
                           │
                           ▼
                        汇总结果 → 回奏
```

关键分水岭：**子任务如果被标注了 `department`，走六部；否则尚书省自己干。**

## 二、部门定义与领域映射

| 部 | 名称 | 领域 | 典型任务 | 主要工具提示 |
|----|------|------|---------|-------------|
| 兵部 | 技术实现 | 代码开发、架构设计、调试优化 | 写模块、修 bug、代码审查、搭建脚手架 | write、edit、git、bash（python/tsc等） |
| 户部 | 数据分析 | 数据处理、量化分析、统计建模 | CSV/JSON 处理、图表生成、爬虫清洗 | python(pandas/matplotlib)、jq、sqlite3 |
| 礼部 | 内容创作 | 文档撰写、文案、翻译、报告 | README、设计文档、汇报、翻译 | write、web_search（查资料） |
| 刑部 | 安全合规 | 安全检查、合规审计、风险前置 | 代码审计、配置检查、依赖漏洞扫描 | fd、grep、bash |
| 吏部 | 协调管理 | 流程管理、任务跟踪、资源协调 | 拆解进度、检查依赖状态、记日志 | write、cron、check_pending_tasks |
| 工部 | 部署运维 | 基础设施、CI/CD、系统配置 | 环境配置、构建脚本、定时任务 | bash、git、cron |

部门以需求为导向，不是每部都必须有对应子任务才执行——一部分在实际使用中就自然而然活跃，其他部可以安静存在。

## 三、新增字段：department

Plan Object 中，中书省规划时给每个子任务标注归属部门：

```yaml
subtasks:
  - id: sub-01
    summary: "数据清洗"
    department: 户部
    prompt: "..."
    ...
  - id: sub-02
    summary: "写分析脚本"
    department: 兵部
    prompt: "..."
    ...
```

一次封驳时中书省可以同时调整 department 分配——如果门下省发现任务的归类不合适。

## 四、部的 prompt 结构（通用模板）

每个部有一份独立的领域提示词，给尚书省派出 subagent 时注入。结构一致：

```
# 部名（领域）

## 角色定位
一句话+2-3条工作原则

## 擅长的工作
该领域常见任务的类型说明，帮助 subagent 判断当前任务是否匹配

## 工具推荐
该领域常用的工具列表及简要使用场景。subagent 按需使用，不限制

## 常见陷阱
该领域容易踩的坑，提醒 subagent 注意
```

**每份预计 150-300 字。** 短而精，不让 subagent 在大量原则描述中淹没核心任务指令。

## 五、派出方式

尚书省是调度者，流程：

```
for each subtask in plan.subtasks:
  if subtask.department 存在:
    读取对应部的 prompt 模板
    构造 subagent 指令：
      [部的领域提示词]
      ---
      子任务 prompt: {subtask.prompt}
      工具提示: {subtask.tools_hint}
      预期产出: {subtask.expected_output}
      ---
      完成信号：输出结果后汇报
    派出 subagent 执行
  else:
    尚书省自行执行
```

串行执行——每个子任务完成后尚书省再看下一个。暂不引入并行依赖拓扑，那是复杂度到位了再考虑的事。

## 六、边界情况处理

| 场景 | 处理方式 |
|------|---------|
| 子任务跨两个部门 | 中书省标注 primary、secondary，尚书省按 primary 派出，secondary 作为参考注入 |
| 某个部门从未被使用 | 不需要有默认行为，只用不被调用 |
| 子任务超简单（如"读取文件"） | 中书省可以不标 department，尚书省自行执行 |
| 子任务阻塞 | subagent 通过 escape hatch 报告阻塞，尚书省看情况重试或标记 blocked |
| 同一子任务需多部门协作 | 第一版暂不支持。复杂的多角色分工留到后续版本 |

## 七、需要你确认的点

1. **六部的领域映射**——这六类的划分跟你实际情况对得上吗？有没有需要调整或合并的？
2. **部门谁分配**——中书省规划时标注，这个方向可行？还是你更希望尚书省执行时自己判断？
3. **子任务超简单不标 department 由尚书省自行执行**——这个逻辑合理吗？
4. **每部 150-300 字的 prompt**——这个深度你觉得够了，还是需要更详细一些？
